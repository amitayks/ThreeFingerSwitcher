## Context

The Files band (`files-band`) gave the launcher a local-only Finder: drill by trackpad, **lift to open**. Opening today is `FileOpenService.prepareOpen(entry) → workspace.open(url)` — it hands the file to another app and pulls the user out of the platform. The user wants media to *play in place* under the same trackpad grammar, and the codebase now has every primitive a player needs:

- **A proven non-activating, interactive overlay.** The AI canvas runs in a `SwitcherPanel` (`.nonactivatingPanel`, never main, key only when `keyInteractive` is flipped for SwiftUI buttons). It sits at `.popUpMenu`, ignores the focus war, and resolves by trackpad. A player that is *purely gesture-driven* needs even less than the canvas — it never needs `keyInteractive`, so it stays fully non-activating.
- **Two proven modal trackpad sub-states.** `GestureRecognizer.feed()` short-circuits into `trackCanvasResolution` (one-shot, `launcherCanvasResolutionActive`) and `trackFilesDrill` (sustained, `filesDrillActive`) as its *first statements*, before the 3-/4-finger latch. The player adds a third sustained one.
- **The converging positional-navigation joystick** (`positional-navigation`, effectively landed): anchored center + footprint scale + per-axis inner/outer zone hysteresis + dwell-eased auto-repeat, with a generalized relative **+1-finger action-menu** intent (D6) and **two-finger** canvas resolution (D5). The player is the natural third surface for this language.
- **The seam discipline.** MLX lives in `GemmaRuntime` behind `LLMRuntime`; `NSWorkspace` behind `FileWorkspace`. Core stays pure and `swift test`-covered against stubs. The player's playback engine follows the same rule.
- **Codable per-file stores.** `FavoritesStore`, `ClipboardStore`, `KeyboardLanguageStore` — the resume store is the same shape.

Constraints: Core stays MLX-free **and** engine-free; the player must not become key/main (B3 focus-vacuum / Stage-Manager landmines); `build-app.sh` must bundle native dylibs into `Contents/Resources/` (the metallib precedent); spec-first; existing tests stay green.

## Goals / Non-Goals

**Goals:**
- A built-in player for **video, audio, and images** opened from the Files band, opt-in (default off), playing on the **current Space**, dismissable back to where you were.
- **One trackpad transport grammar** continuous with the rest of the app: 2 = act-within (scrub/volume/play-pause), 3 = action menu, 4 = dismiss platform.
- **Dual engine behind one `MediaPlaybackEngine` seam, both in v1:** AVFoundation default; libmpv the alternative, auto-offered on a decode failure and selectable on demand.
- **Per-file resume** of position + tracks + volume + rate.
- A **pure, `swift test`-covered Core** (transport state machine, scrub/volume/resume math, media-kind classification, fallback decision, action-menu model, state store) against a `StubPlaybackEngine`; only the rendering conformers need `xcodebuild`.
- The player **owns the trackpad** while open (3 fingers = its action menu, not the global switcher) — and **never becomes key/main**, sidestepping the focus war.

**Non-Goals (v1):**
- File operations (move/rename/delete/trash/tag) — navigation/open/play only.
- Remote / network / streaming sources, casting, AirPlay.
- Any playlist / queue / autoplay-next beyond **per-file resume** (a "Continue watching" band is a deliberate follow-up).
- Converting the window switcher to the joystick, or any change to opening/activation latches — untouched, as in `positional-navigation`.
- No new permission, no native-gesture relocation, no re-login.

## Decisions

### D1 — A `media-player` capability owns the domain; three existing specs get deltas
The new `media-player` capability owns the opt-in + open-routing contract, media-kind classification, the engine seam + both conformers + the fallback rule, the transport state machine, the resume store, the action menu, the image viewer, the player surface, and the error taxonomy. Deltas go only to `gesture-recognition` (the recognizer mechanism), `configuration-hub` (the Player page), and `tunable-settings` (opt-in + tunables). **The Files-band open branch and the player surface are NOT `files-band`/`launcher-overlay` spec deltas:** the player is a *separate* overlay (the launcher dismisses on lift-to-open), and `files-band` is itself an unarchived change. *Alternative:* delta `files-band`/`launcher-overlay`. Rejected — it couples to specs not yet in `openspec/specs/`, and `positional-navigation` set the precedent of touching the files drill in code while declaring deltas only against landed specs.

### D2 — Dual engine behind `MediaPlaybackEngine`, AVFoundation default + libmpv fallback, both v1
`MediaPlaybackEngine` is a protocol (Core, MLX/AppKit-light): `load(url)`, `play/pause`, `seek(by:)`/`seek(to:)`, `setRate`, `setVolume`, `audioTracks`/`subtitleTracks` + `select(track:)`, observable `position`/`duration`/`isPlaying`, and a `canDecode`/load-failure signal. Two conformers ship: **`AVPlaybackEngine`** (default — `AVPlayer`/`AVPlayerLayer`, hardware decode, container-carried subtitle/audio tracks) and **`LibmpvEngine`** (libmpv — plays anything, external `.srt`). The app injects the engine factory at the seam in `main.swift` (exactly where the real `LLMRuntime` is injected). Tests run against **`StubPlaybackEngine`** so all transport logic is `swift test`-covered. *Alternative:* AVFoundation-only with a later seam, or libmpv-only. Rejected — the user locked both for v1, and the seam keeps Core engine-free either way.

### D3 — AVFoundation→libmpv fallback is an observable decision, not a silent retry
When the default engine reports it cannot decode (load failure / unsupported codec), the player does **not** silently fail or silently swap: it transitions to a bounded, non-blocking **`.unsupported(byDefault)`** state offering **"Open in libmpv"** (commit) or dismiss — the same observable-failure law as `FileOpenService`/`AICommandExecutor`. The libmpv path is also reachable on demand from the action menu regardless of decode success (power-user override). The fallback *decision* (given a decode-failure signal + which engines exist, what to offer) is a **pure function** in Core, unit-tested. *Alternative:* auto-swap to libmpv transparently. Rejected — a transparent swap hides which engine is running (matters for the action menu's "open in libmpv" and for diagnosing playback quality); the offer is one trackpad commit away.

### D4 — A third sustained recognizer sub-state: `playerActive` → `trackPlayer`
Mirror `filesDrillActive`/`trackFilesDrill` exactly: `playerActive` is a `Bool` whose `didSet` seeds a fresh session, and `trackPlayer(frame)` is routed as a **first-statement bypass** in `feed()` (before the 3/4 latch). While `playerActive`, **all** finger activity is the player's — so a 3-finger count is the action menu, never the global switcher. The transport intents (`playerSeek(dir)`, `playerVolume(dir)`, `playerTogglePlayPause`, `playerActionMenu`, `playerSelectMenuItem`, `playerDismiss`) are new delegate methods with **default no-op extensions** (so existing delegates compile). Re-baseline the anchor on **every contact-count change** (the documented landmine) so a leaving finger fires no phantom step. *Alternative:* nest the player under the launcher controller. Rejected — the launcher dismisses on lift-to-open, so the player must be its own top-level surface and sub-state.

### D5 — Transport is the positional joystick, auto-repeating on *both* axes
The player routes through the same `PositionalNavigator` core: 2-finger horizontal = `playerSeek` (out-and-back = one step; hold = dwell-eased auto-repeat = fast-forward/rewind), 2-finger vertical = `playerVolume` (same). **Unlike Files/Clipboard (which suppress horizontal auto-repeat for deliberate depth/pin), the player auto-repeats on both axes** — a per-surface per-axis repeat config the positional model already supports. A **2-finger tap** (contact with no excursion past the inner deadzone, then lift) toggles play/pause. The **3-finger** relative +1 raises the action menu; **lift on a row** selects it (the Open-With scrubbable-picker pattern, verbatim). **4 fingers** dismiss. *Alternative:* absolute "dance-pad" position-scrub (finger X on pad = position in clip). Rejected for v1 — it's a different interpretation layer; stepwise-seek-with-repeat is continuous with every other surface (recorded as a follow-up).

### D6 — A purely gesture-driven, non-activating, full-screen player surface
`PlayerOverlayController` owns its own `SwitcherPanel` (reused class), sized to the full screen, at `.popUpMenu` with `[.canJoinAllSpaces, .fullScreenAuxiliary, .ignoresCycle]` — **never** `keyInteractive`, **never** key/main. Because every control is a trackpad intent (no SwiftUI buttons to click), the player needs none of the canvas's key-window concession, so it stays clear of the focus-vacuum and Stage-Manager singleton landmines entirely. `PlayerView` is pure presentation over the engine's observable state + a thin host for the engine's render layer (`AVPlayerLayer` / libmpv's layer), injected so Core/the view stay engine-free. Entrance/teardown use the `BubbleMorph` law (the app's spring); teardown stays **synchronous** on Space-switching dismiss (the files-band ghost-on-Space-switch landmine). *Alternative:* a real key `AVPlayerView` window. Rejected — it breaks the app's never-become-key invariant and re-opens the focus war for no gain, since the controls are gestures.

### D7 — Per-file resume via a Codable `PlaybackStateStore`, keyed by a stable file identity
`PlaybackStateStore` persists `{ resumePosition, audioTrackID, subtitleTrackID, volume, rate, lastOpened }` per file, keyed by a **stable identity** (absolute path, with a content-size/mtime tiebreak so a moved-but-identical file still resolves; full content hashing deferred as too costly for large media). On open, if a saved position exists **past a resume threshold and not near the end**, playback resumes there; otherwise from the start. The store enforces a bounded entry cap (LRU by `lastOpened`) so it can't grow without limit — the `ClipboardStore` retention pattern. *Alternative:* content-hash keys. Rejected for v1 — hashing multi-GB files on open is too slow; path+size+mtime is good enough and cheap.

### D8 — Media-kind classification is pure UTI logic in Core
`MediaKind.classify(FileEntry) -> MediaKind?` (`.video`/`.audio`/`.image`/`nil`) via UTType conformance (`.movie`/`.audio`/`.image`), so the open branch and the per-kind Hub toggles share one testable predicate. `FileOpenService.prepareOpen` branches on `MediaKind.classify(entry) != nil && settings.useBuiltInPlayer && settings.builtInPlayerHandles(kind)` → hand to `PlayerController.play(entry, kind:)` instead of `workspace.open`; the defusable `PendingOpen` lifecycle is preserved (a four-finger discard before the fuse opens nothing). Open-With (+1 lift) always opens externally, untouched. *Alternative:* engine-driven "can you play this?" classification. Rejected — that needs an engine instance just to route; UTI classification is pure, instant, and testable, with the *decode* check happening later (D3) as the fallback trigger.

### D9 — Errors map at the boundary into a `MediaPlayerError` taxonomy, surfaced bounded + non-blocking
A Core `MediaPlayerError: LocalizedError` taxonomy parallel to `FileActionError`/`RuntimeError` (cases: `loadFailed`, `unsupportedByDefaultEngine`, `engineUnavailable`, `decodeFailed`, …), each with a clean per-case headline + opt-in copyable `details`. Engine/OS errors (AVFoundation `NSError`, libmpv error codes) are mapped **at the conformer boundary** so Core never sees vendor types. A failure is **observable state** (`.failed(headline:details:)`) rendered as a bounded, non-blocking card over the player (clean headline, capped length, copyable details behind a disclosure, Retry/Dismiss/Open-in-libmpv) — **never** an `NSAlert`, never raw error text in a headline. Cancellation/dismiss is not a failure.

### D10 — libmpv bundled like the metallib; gated so its absence degrades, not crashes
libmpv ships as a **bundled dylib** copied into `Contents/Resources/` by `build-app.sh` (the metallib precedent: miss it and the player is SIGKILL'd at first decode). The `LibmpvEngine` is constructed lazily and its availability probed at injection; if the dylib is missing/unloadable the engine reports **unavailable** and the player degrades to AVFoundation-only with an observable message — never a hard crash. `swift build`/`swift test` never link libmpv (it lives in the app/conformer path); the app builds via `xcodebuild`. *Alternative:* `Process`-spawn the `mpv` CLI. Rejected — no in-overlay rendering, no frame-accurate control, and it would launch an external window (the teleport we're removing).

## Risks / Trade-offs

- **Full-screen video compositing in a non-key `.canJoinAllSpaces` borderless panel** → *Mitigation:* AVPlayerLayer should compose fine (it's a CALayer); libmpv's own GL/Metal layer is the riskier one — spike it before wiring (and it's the *fallback* engine, so AVFoundation proves the surface first). Flagged as the top in-hand validation.
- **2-finger scrub vs. incidental two-finger scroll/resolve** → *Mitigation:* the player owns the trackpad (the scroll tap already mediates two-finger scroll while an overlay is open); reuse the positional inner/outer hysteresis so a small nudge-and-return is one seek step, not noise. Tune in-hand.
- **libmpv dylib packaging/signing** (notarization of a bundled native lib) → *Mitigation:* treat exactly like the MLX metallib — copy into `Resources/`, sign as part of the bundle, verify the release workflow staples it. Documented as a build landmine.
- **Resume identity for moved/renamed files** → *Mitigation:* path+size+mtime key; a moved file just starts fresh (acceptable). No silent wrong-file resume because size+mtime must match.
- **2-finger tap = play/pause vs. lift-commit muscle memory** → *Mitigation:* in the player there is nothing to "commit," so a tap is free; confirm the tap threshold (no excursion past inner deadzone) doesn't fire on a tiny scrub. In-hand.
- **Regressing the load-bearing recognizer** → *Mitigation:* `playerActive`/`trackPlayer` is purely additive (a new first-statement bypass + no-op-default delegate methods); the latch, relax-to-two, and re-baseline rules are untouched; existing recognizer tests stay green.
- **Stale `CLAUDE.md` landmines** → *Mitigation:* add a Files/Player landmine: the player owns the trackpad via a third modal sub-state, is non-activating (never key), and `build-app.sh` must bundle the libmpv dylib.

## Migration Plan

Purely **additive** and **opt-in (default off)**: no data migration, no persisted-favorites schema change (resume state is its own store), no permission, no native-gesture relocation, no re-login. Settings written before this change load with the opt-in OFF (the legacy-load scenario). **Sequencing:** land after `positional-navigation` and `files-band` archive (both effectively complete) so the joystick grammar and the open seam exist. **Rollback** is turning the opt-in off (media files revert to opening in the system app) or reverting the change; nothing persists that outlives it. Verify Core logic via `swift build`/`swift test` against `StubPlaybackEngine` (transport state machine, scrub/volume/resume math, media-kind classification, fallback decision, action-menu model, state store); compile-verify the AVFoundation + libmpv conformers and the app via `xcodebuild`; in-hand feel + compositing validation on a stable-signed build (agent can't sign). The MLX/`GemmaRuntime` split is untouched.

## Open Questions

- **libmpv distribution** — vendor a prebuilt `libmpv.dylib`, build it in CI, or pull via a package? Pick the path that notarizes cleanly with the existing release workflow.
- **Seek step defaults & scaling** — fixed ±10 s, or scale the step to clip duration (e.g. % of total) so a 3-hour film and a 30-s clip both feel right? Default + tune in-hand.
- **2-finger tap detection** — exact excursion/time window that distinguishes a play/pause tap from a micro-scrub. Tune in-hand.
- **Image viewer transport** — images have no timeline: does 2-finger horizontal pan (when zoomed) / next-image, vertical zoom? Decide the image gesture map when wiring the viewer (it shares the surface + action menu but not the seek/volume axes).
- **Subtitle source for AVFoundation** — container-carried only in v1, with external `.srt` deferred to the libmpv path? Confirm.
- **Resume affordance** — auto-resume silently, or show a brief "Resume / Start over" choice on open (an action-menu entry vs. a transient prompt)?
